# Internationalization (i18n)

ZinharCMS uses a small first-party i18n layer in the React admin panel. The goal is to keep locale handling explicit, typed, and easy to extend without coupling UI copy to components.

## Current Locales

| Code | Language | Direction |
| --- | --- | --- |
| `en` | English | `ltr` |
| `fa-IR` | Persian | `rtl` |

Locale codes follow BCP 47. Use full regional codes when language behavior, formatting, or direction can differ by region. Persian is registered as `fa-IR`.

## Files

```text
frontend/src/i18n/locales.ts          Supported locales, fallback, direction, detection
frontend/src/i18n/messages.ts         Typed translation dictionaries
frontend/src/i18n/I18nProvider.tsx    React provider, persistence, document lang/dir sync
frontend/src/i18n/LanguageSelect.tsx  Reusable locale switcher
frontend/src/i18n/index.ts            Public exports
```

## Runtime Behavior

- The app reads the saved locale from `localStorage` key `zinhar.locale`.
- If there is no saved locale, the browser locale is normalized.
- Unsupported browser locales fall back to `en`.
- `fa` and `fa-*` normalize to `fa-IR`.
- `en-*` normalizes to `en`.
- The provider updates `<html lang="..." dir="...">` whenever the locale changes.
- Missing translation keys fall back to English.

## Adding a New Language

1. Add a locale definition in `frontend/src/i18n/locales.ts`:

```ts
{
  code: "de-DE",
  name: "German",
  nativeName: "Deutsch",
  direction: "ltr",
}
```

2. Add normalization rules if the generic language code should map to the regional code:

```ts
if (language === "de") return "de-DE";
```

3. Add a matching dictionary in `frontend/src/i18n/messages.ts` with the same keys as `en`.

4. Use `useI18n()` in components and call `t("message.key")` instead of hard-coded UI text.

5. For text with values, use interpolation:

```tsx
t("dashboard.version", { version: health.version })
```

6. For layout direction, prefer logical CSS properties like `margin-inline-start`, `border-inline-end`, and `padding-inline-start`.

## Key Naming

Use domain-based keys:

```text
app.nav.dashboard
auth.submit.login
dashboard.metric.contentTypes
```

Avoid storing business data, API enum values, or user-generated content inside translation dictionaries. Only UI copy belongs in `messages.ts`.

## RTL Guidance

Persian runs with `dir="rtl"`. Keep components direction-safe by using:

- `inline-start` and `inline-end` instead of left/right for new CSS
- `text-align: start` where possible
- icon-only buttons with translated `aria-label`
- translated visible labels for form controls and navigation

## Current Coverage

The i18n layer is wired into the app root, authentication screen, main shell/navigation, language switcher, and dashboard. Other admin pages can be migrated incrementally by moving hard-coded labels into `messages.ts` and replacing them with `t(...)`.
